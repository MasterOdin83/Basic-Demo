Ballastlane - Technical Exercise

A small app, shipped like a real product

A task manager built end to end — versioned requirements, clean architecture, TDD, security review and cloud deployment — to show the process, not just the code.

View the source on GitHub ↗

Stack

Angular 21 front end (zoneless, signals). Two ASP.NET Core (.NET 10) Web APIs — a token service (register / login / me) and a JWT-protected tasks API — over EF Core + SQLite, each self-documenting with OpenAPI + Scalar (tasks API docs ↗ · auth API docs ↗).

Architecture

Clean Architecture: a dependency-free core owns the business rules (input validation, per-user task ownership, PBKDF2 password hashing); storage is swappable behind repository interfaces.

Quality — TDD

38 automated tests: 32 xUnit on the back end (services with fakes, repositories on in-memory SQLite, endpoints via WebApplicationFactory) plus 6 Vitest UI tests. All green before every merge.

SDLC

Versioned requirements with acceptance criteria → implementation → code review → QA against the acceptance criteria → OWASP security review → CI/CD with GitHub Actions deploying to Azure (Static Web Apps + App Service).

Security

Baked into the SDLC, not bolted on: every input is validated server-side against parameter tampering, the tasks API requires JWT bearer tokens with per-user ownership checks, registration never reveals whether a username exists, and the deployed site serves hardened security headers rated A+ ↗.

Built with GenAI

Pair-built with Claude Code driving a multi-agent pipeline: requirements analyst, UI and API developers, code reviewer, QA tester, OWASP security and DevOps agents. Every generated change is gated by tests and review — including temporarily dropping a dependency over an unpatched CVE and restoring it once a safe version shipped.

Exercise brief → implementation

Requirements coverage

Every requirement from the .NET technical interview brief, mapped to where it lives in this solution.

  • Candidate-authored user story "As a registered user, I want to create, view, edit and delete my personal tasks (title, description, status, due date) so I can keep track of my work. Only I can see and manage my own tasks."
  • Database: app table + users table, PK and 2+ fields SQLite — Tasks (Id PK, Title, Description, Status, DueDate, UserId FK) and Users (Id PK, Username unique, PasswordHash), auto-created and seeded on first run.
  • Web API with CRUD endpoints Basic.API — GET / POST / PUT / DELETE on /api/tasks with proper verbs, parameters and status codes; OpenAPI docs served via Scalar ↗.
  • Second API: create user, login, authorized and anonymous endpoints BasicSTS.API — POST /api/auth/register, POST /api/auth/login (JWT), GET /api/auth/me [Authorize]; registration replies created-or-not only, so usernames can't be enumerated.
  • Data access layer Basic.Data — EF Core + SQLite repositories implementing the core's interfaces.
  • Independent business logic layer Basic.Core — zero dependencies; owns validation, per-user ownership and password rules.
  • Unit tests for all components 32 xUnit tests across data access, business logic and API endpoints, plus 6 Vitest UI tests.
  • Responsive, user-friendly frontend with full CRUD Angular 21 — landing page with slide-in login/register drawer, guarded routes, JWT interceptor, inline task editing, clean component and service structure.
  • README with setup docs, seed data and credentials Step-by-step setup, endpoint tables, and a seeded demo account with three tasks.
  • GenAI deliverable The brief's GenAI scenario (a task-management API) is this app itself, pair-built with Claude Code; the prompt and validation write-up is in progress.
